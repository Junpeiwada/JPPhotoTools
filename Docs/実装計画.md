## 完了済みの実装（PH0 / PH1 / PH2）

### 成果物

```
JPPhotoTools/
├─ App/                      統合アプリ（XcodeGen）
│  ├─ project.yml            3 パッケージを path 参照
│  └─ Sources/
│     ├─ JPPhotoToolsApp.swift  @main。TabView ルート＋比較ビューワ Window シーン
│     ├─ ContentView.swift      3 タブの TabView 本体

│  ├─ GainForgeApp/          GainForge の App 層をライブラリ化
│  ├─ JpegResizerApp/        JpegResizer の App 層をライブラリ化
│  └─ RawTrashCore/          RAW-Trash の Swift 移植（テスト 6 件）
└─ External/
   ├─ GainForge/             本家ソースの丸ごとコピー（GainForgeCore を path 参照）
   └─ JpegResizer/           本家ソースの丸ごとコピー（JpegResizerCore を path 参照）
```

### 検証済みの事実
- `xcodebuild` で統合アプリがビルド成功し、3 タブ切り替えでアプリが起動する。
- ML-LUT（`apple-gainlut-17-perchan.bin`, 78,608 byte）が `GainForge_GainForgeCore.bundle`
  として統合 .app に同梱され、`Bundle.module` 解決が保たれる（`.hdrML` が降格しない）。
- `RawTrashCore` はユニットテスト 6 件通過（2 パス仕分け・連番退避・大小無視ペア判定・
  RAW/ 連番衝突・空フォルダ・不在フォルダ）。
- code-reviewer の独立レビューで総合判定 Ship 可（重大バグなし）。指摘 M-1（copy+remove
  フォールバック時の複製＆エラー握りつぶし）は修正済み。

### 既知の負債（PH1.5 以降で解消）
- **変換系 2 タブの App 層が重複**。`AppViewModel` / `FileItem` / `StatusBarView` などが

  重複して抱えている（共通化で 1 本化予定）。

## PH1.5 — 共通シェル抽出（完了）

### 実装結果（サマリ）
変換系 2 タブを共通シェル `PhotoKitShared` に統合した。最終的な形:

- **`FileItem<Extra>`**: 共通コア＋機能固有部 `extra`（GainForge: `GainForgeExtra{gainMap}` /
  JpegResizer: `JpegResizerExtra{pixelSize, outputPixelSize}`）。
- **`protocol FeatureShell`**: 機能差分（入力収集・probe・変換・出力計画・大小無視フラグ）の注入点。
  値型（struct）で `GainForgeFeature` / `JpegResizerFeature` が実装。
- **`open class AppViewModel<F: FeatureShell>`**: ドロップ・probe スロットル・スライディング
  ウィンドウ並列変換・上書き確認・集計の骨格。各機能の ViewModel が継承し、設定の `@Published`
  保持と `makeSettings()` / `willStartConversion(extras:)`（GainForge の ML 降格通知）だけを足す。
- **`OutputPlanner`**: 出力先ディレクトリ／ファイル名をクロージャ注入で共通化。`caseInsensitive`
  フラグで APFS 大小無視の衝突回避を機能ごとに切替（JpegResizer=true / GainForge=false）。
- 共通 UI 部品（`StatusBarView` は各モジュール、`FlowLayout` / `Chip` / `ImageProbe` /
  `WindowFrameAutosave` は PhotoKitShared）も 1 本化。

### 品質保証
独立レビュー（code-reviewer）を 2 回実施し、挙動リグレッションを 2 件検出・修正した:
1. **ML/LUT 降格通知のタイミング**: `convertOrCancel` override（出力計画・上書き確認の前）で
   通知していたのを、`willStartConversion` フック（`orderedIDs` 確定後）へ移設。上書きキャンセル時に
   通知が出ない・`didNotifyMLFallback` を誤消費しない旧挙動を回復。
2. **OutputPlanner の大小文字**: JpegResizer が移行前に持っていた APFS 大小無視の衝突回避が
   共通化で失われていた（`Photo.jpg`/`photo.jpg` 取り違え・上書き事故）。`caseInsensitive` で回復し、
   `PhotoKitSharedTests/OutputPlannerTests`（4 件）で固定。
3. probe の遅延着弾が変換確定行の `extra` を潰す問題を、`applyProbe` を `.waiting` 行限定にして解消。

以下は着手時の計画（記録として残す）。

### ゴール
変換系 2 タブ（GainForge / JpegResizer）の重複した App シェルを `PhotoKitShared` パッケージへ
昇格し、タブごとに差し替えるのは **①変換関数 ②設定 UI ③テーブル列** の 3 点だけにする。
RAW-Trash は仕分け系でパイプラインが異なるため共通シェルには乗せない（現状の専用タブ維持）。

### なぜ慎重にやるか
`AppViewModel`（GainForge 556 行）は「純粋な骨格」ではなく、GainForge 固有ロジック
（既存 HEIC 検出 `ExistingOutputFinder`・ML/LUT 降格通知・出力先モード）が骨格に織り込まれ、
かつ機能固有フィールド（`gainMap` / `pixelSize`）が `FileItem` に直接混入している。ここを
壊すと「触ると壊れる」生転写・HDR 合成の変換経路（GainForge CLAUDE.md が警告）に波及し得る。

### 共通シェルと差分の境界（調査で確定済み）

| 要素 | 共通化する | タブ固有（差し替え） |
|------|-----------|---------------------|
| probe スロットル / スライディングウィンドウ並列 / `OutputPlan` / 上書き確認 / `ConversionOutcome` | ✅ 一字一句ほぼ同一 | — |
| `StatusBarView` | ✅ 完全一致 | — |
| `FooterBarView` / `WindowFrameAutosave` / `FlowLayout` | ✅ 実質一致 | — |
| `FileItem` の共通コア（id/url/thumbnail/status/bytes/outputURL/errorMessage） | ✅ | — |
| `FileItem` 固有部 | — | `gainMap`(GF) / `pixelSize`,`outputPixelSize`(JR) |
| `RowStatus` | 共通ケース | `.existing`(GF のみ) |
| probe 内容 | — | hasGainMap+既存検出(GF) / pixelSize(JR) |
| 設定 | quality / resize 系 | sdrMode+outputMode(GF) / なし(JR) |
| `convert(...)` 呼び出し | — | `GainForge.convert` / `JpegResizer.convert` |
| `OutputPlanner` | — | App 層・.heic・outputMode 対応(GF) / Core 層・.jpg(JR) |
| テーブル列 | サムネ/名前/サイズ | ゲインマップ列(GF) / 寸法列(JR) |

### 型抽象化の方針（決定済み）
`FileItem<Extra>` **ジェネリクス**で共通コアを基底にし、機能固有部を `Extra` 型パラメータで
持たせる（型安全優先）。`AppViewModel` もジェネリック化する。

### 作業ステップ（案）
1. `PhotoKitShared` パッケージを新規作成。まず無条件に共通な型を移動:
   `AppPhase` / `OverwriteResolution` / `FlowLayout` / `Chip` / `SizeFormat` /
   `StatusBarView` / `FooterBarView` / `WindowFrameAutosave` / `ImageProbe`。
2. `FileItem<Extra: Equatable & Sendable>` を定義（共通コア＋`extra: Extra`）。
   `RowStatus` は共通ケース＋機能拡張の合成方式を決める（`.existing` の扱い）。
3. `AppViewModel<Extra>` を抽出。probe とバッチ変換の骨格を移し、機能差分を
   「probe クロージャ」「変換クロージャ」「設定」の注入点として切り出す。
4. `OutputPlanner` の相違（拡張子・outputMode）を吸収するインターフェースを決める。
5. GainForge タブ / JpegResizer タブを共通シェル上に再実装し、差し替え 3 点だけ持たせる。
6. 各機能の既存 App テスト（`AppViewModelTests`）を回帰テストとして通す。**1 タブずつ**
   移行し、都度ビルド＋起動＋テストで確認する（一括置換しない）。

### 完了条件
- 3 タブすべてビルド成功・起動、GainForge の生転写/HDR 合成/ML 降格が従来通り動く。
- 変換系 2 タブの `AppViewModel` 実装が 1 本（ジェネリック）に統合され重複が消える。
- 既存の回帰テストが緑。
