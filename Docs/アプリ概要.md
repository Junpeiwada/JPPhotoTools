## 1. 目的

撮影後の写真処理は「取り込み → 整理 → 変換 → 書き出し」という一本の流れなのに、
機能ごとにアプリが分かれていて、フォルダを開き直しながら渡り歩く必要があった。
JPPhotoTools は、この後工程を**1つのアプリ内でタブを切り替えて連続処理**できるようにする。

最大の価値は「**ドロップしたフォルダをタブ間で引き継げる**」こと。一度読み込んだ写真群を、
整理 → 変換 → リサイズと、アプリを閉じずに流していける。

## 2. 統合対象

| 由来アプリ | 機能 | 元スタック |
|-----------|------|-----------|
| **RAW-Trash** | RAW/JPG ペアを仕分け、孤立 RAW を `Del/` へ退避 | Electron / TypeScript |
| **GainForge** | ゲインマップ付き HEIC を生成（SDR→HDR 変換） | Swift / SwiftUI |
| **JpegResizer** | 共有用に高品質 JPEG リサイズ | Swift / SwiftUI |

GainForge と JpegResizer は「Core + CLI + App（XcodeGen）」の同一テンプレートで作られた
双子アプリ（JpegResizer は GainForge をコピー元にしたもの）。この2つを**土台**にし、
Electron 製の RAW-Trash を Swift へ移植して取り込む。

> 補足: 調査時点では GeoTagger（ジオタグ）・UniPic（重複検出）も統合候補だったが、
> 今回のスコープからは外し、まず上記3つで統合の骨格を確立する。両者は
> [将来の拡張候補](#8-将来の拡張候補)として設計上の受け皿だけ用意しておく。

## 3. ワークフローとタブ構成

タブは撮影後の処理を**時系列順**に並べる。

```
① 取り込み・整理   RAW-Trash    RAW/JPG ペア判定 → RAW/ JPG/ Del/ に仕分け
        ↓
② HDR 変換        GainForge    ゲインマップ付き HEIC を生成
        ↓
③ リサイズ書き出し JpegResizer  共有用に高品質 JPEG へ縮小
```

ドロップ済みのファイル群は共通の状態（後述の共有 App シェル）に保持され、
タブを切り替えても保たれる。

## 4. アーキテクチャ

### 4.1 3層構成の踏襲

GainForge / JpegResizer が採用する3層をそのまま拡張する。

- **Core**: 変換ロジック（UI 非依存・テスト可能な Swift Package ライブラリ）
- **CLI**: Core を叩く薄いラッパ（ArgumentParser、動作確認用）
- **App**: SwiftUI アプリ（XcodeGen の `project.yml` で管理、Core を `path` 参照）

設計原則（GainForge の CLAUDE.md 由来）:
**「CLI と GUI に変換ロジックを持たせない。新しい変換挙動は必ず Core に実装し、両 UI から呼ぶ」**
を統合後も厳守する。共通シェルからは各機能の `convert(...)` 越しにしか変換に触らない。

### 4.2 レイヤ設計

```
┌─ タブ層（機能の切り替え） ────────────────────────────┐
│  TabView / NavigationSplitView で機能を選択            │
│  差し替えるのは ①変換関数 ②設定UI ③テーブル列 の3点だけ │
├─ 共通 App シェル（1つに昇格） ────────────────────────┤
│  AppViewModel / ドロップ→probe パイプライン /          │
│  バッチ並列エンジン / FileTableView / FlowLayout・Chip │
│  / StatusBar・Footer / Sparkle 自動更新                │
├─ 機能 Core（Swift Package・UI 非依存・横並び） ────────┤
│  RawTrashCore  GainForgeCore  JpegResizerCore          │
│  └ 共通型は PhotoKitShared に集約                       │
└────────────────────────────────────────────────────┘
```

- **機能 Core は融合させず横並び**にする。統合アプリの `project.yml` で各パッケージを
  並列依存させ、Core 同士は独立を保つ（GainForge の生転写経路など、触ると壊れる
  ロジックを各 Core に閉じ込めるため）。
- 重複する共通型（`ConversionResult` / `ResizeMode` / `OutputPlanner` / `ResizePlanner`）は
  **`PhotoKitShared`** に切り出し、各機能 Core が依存する。

### 4.3 タブ切り替えで差し替える3点

バッチ並列エンジンは変換の中身を知らなくてよい。タブごとに差し替えるのは以下だけ:

1. **変換関数** — `(input, output, settings) throws -> ConversionResult` を満たす
   クロージャ、または `protocol PhotoOperation`
2. **ツールバーの設定セクション** — 各機能の設定 View（品質スライダー、
   リサイズ方式、SDR 方式など）を共通の `FlowLayout` に差し込む
3. **テーブル列定義** — GainForge は HDR/ゲインマップ列、JpegResizer は寸法列、
   RAW-Trash は分類結果列、と機能ごとに異なる

### 4.4 共通化の根拠

調査で、GainForge と JpegResizer の App 層はほぼコピー状態であることを確認済み:

- `StatusBarView` は**バイト単位で完全一致**
- `AppViewModel`（ドロップ → probe → スライディングウィンドウ方式のバッチ並列変換）、
  `FlowLayout`（自作 SwiftUI Layout）、`Chip`、`FooterBarView`、`WindowFrameAutosave` が実質共通
- JpegResizer は `OutputPlanner` を App 側から Core に昇格させた以外、GainForge の
  画面設計・状態遷移をそのまま踏襲

したがって統合は「**一度やった View 差し替えを、タブで束ねるだけ**」に近い。
共通シェルを1つへ昇格させて重複を消すことが、最初の最大の利得になる。

### 4.5 実装状況（PH1）

PH1 では、上記「共通シェル1つへの昇格」を**一足飛びにはやらず**、まず各機能の App 層を
モジュール分離して TabView で束ねる形で統合の骨格を作った。実際の構成:

```
JPPhotoTools/
├─ App/                      統合アプリ（XcodeGen）
│  ├─ project.yml            3パッケージを path 参照
│  └─ Sources/
│     ├─ JPPhotoToolsApp.swift  @main。TabView ルート＋比較ビューワの Window シーン
│     ├─ ContentView.swift      3タブの TabView 本体
│     └─ RawTrashTab.swift      取り込み・整理タブ（RawTrashCore を直接使用）
├─ Packages/
│  ├─ GainForgeApp/          GainForge の App 層をライブラリ化（public: GainForgeTab / ViewerModel / ViewerView）
│  ├─ JpegResizerApp/        JpegResizer の App 層をライブラリ化（public: JpegResizerTab）
│  └─ RawTrashCore/          RAW-Trash の Swift 移植（新規・テスト付き）
└─ External/
   ├─ GainForge/             本家ソースの丸ごとコピー（GainForgeCore を path 参照）
   └─ JpegResizer/           本家ソースの丸ごとコピー（JpegResizerCore を path 参照）
```

- **名前空間の衝突回避**: GainForge / JpegResizer の App 層は `AppViewModel` / `FileItem` /
  `ContentView` など**同名の型**を持つため、同一アプリターゲットに全ソースを入れると多重定義に
  なる。各 App 層を独立 Swift Package（`GainForgeApp` / `JpegResizerApp`）にラップし、
  **モジュール境界で名前空間を分けて**解消した。統合アプリは各モジュールが公開する
  「タブ用エントリ View」（`GainForgeTab` / `JpegResizerTab`）だけを import して使う。
- **公開は最小限**: 各モジュールで public にしたのはタブ View とその `init` のみ（GainForge は
  比較ビューワ共有のため `ViewerModel` / `ViewerView` も public）。内部型（`FileItem` 等）は
  internal のまま。元 App の `@main` 構造・Sparkle（Updater/AboutView）はタブ本体に不要なため
  除外した（統合アプリの自動更新は PH3 の配布フェーズで別途）。
- **比較ビューワ**: GainForge のビューワは元設計どおり**別ウィンドウ**で開く。共有状態
  `ViewerModel` は統合アプリのルート（`JPPhotoToolsApp`）で 1 つ保持し、GainForge タブと
  ビューワ Window シーンの双方へ `environmentObject` 注入して同一インスタンスを共有する。
  行のダブルクリックでモジュール内から `openWindow(id: GainForgeTab.viewerWindowID)` が呼ばれる。
- **ビルトイン化の維持**: ML-LUT（`apple-gainlut-17-perchan.bin`）は App 層モジュール経由でも
  `GainForge_GainForgeCore.bundle` として統合 .app に正しく同梱され、`Bundle.module` 解決が
  保たれることを確認済み（`.hdrML` が降格しない）。

## 5. 移植の勘所

RAW-Trash の中核ロジックは小さく（実質 80 行程度）、外部依存もない。

| 項目 | 元実装（Electron） | Swift 置換先 | 難易度 |
|------|-------------------|-------------|--------|
| ペア判定・フォルダ移動 | `fs` / `path` | `FileManager` | ★☆☆☆☆ |

- 拡張子（`.arw` `.orf` `.cr3` `.cr2`、大小問わず）で RAW を判別し、同名 JPG の有無で
  孤立 RAW を判定 → `Del/` へ、残りを `RAW/` `JPG/` に仕分けるだけ。
- 画像デコードも EXIF 読み書きも不要。`FileManager` の
  `contentsOfDirectory` / `moveItem` / `createDirectory` で 1:1 移植できる。
- クロスボリューム移動時は `moveItem` 失敗 → copy + remove のフォールバックを踏襲。

GainForge / JpegResizer は既存 Swift 実装をそのまま Core として取り込むため移植は不要。
ML-LUT リソース（`Bundle.module` 依存）解決のため、**GainForgeCore はパッケージのまま保つ**
（App ターゲットに溶かさない）。

## 6. 進め方（フェーズ計画）

> 実装で方針を一部見直した。当初 PH1 で「共通シェル抽出（PhotoKitShared 昇格）」を
> 先にやる計画だったが、GainForge / JpegResizer の `AppViewModel` を精読した結果、
> パイプライン骨格は一致するものの、GainForge 固有ロジック（既存 HEIC 検出・ML/LUT
> 降格通知・出力先モード）と機能固有の型フィールド（`gainMap` / `pixelSize`）が骨格に
> 織り込まれており、堅牢なジェネリック共通シェル化は数百行規模で**動いている変換経路を
> 壊すリスクが高い**と判断。そこで「先に動く 3 タブ統合」を PH1 とし、共通シェル抽出を
> PH1.5 に分離した。

- **PH1 — App 層モジュール分離＋ TabView 統合（実装済み）** ✅
  各機能の App 層を独立ライブラリ化して名前空間衝突を解消し、TabView で 3 タブに束ねた。
  「起動して切り替えれば各機能が動く」統合の骨格が完成。RAW-Trash は仕分け系で変換系と
  パイプラインが異なるため、共通シェルに乗せず専用タブ（`RawTrashTab`）とした。
  詳細は [4.5 実装状況](#45-実装状況ph1) を参照。

- **PH1.5 — 共通シェル抽出（未着手・次フェーズ）**
  変換系 2 タブ（GainForge / JpegResizer）の `AppViewModel` を `FileItem<Extra>`
  ジェネリック基盤へ統合し、コピペ状態の App シェルを `PhotoKitShared` へ昇格する。
  機能固有部（変換関数・設定 UI・テーブル列）だけをタブで差し替える理想形にする。
  壊れやすい変換経路のリファクタになるため、回帰テストを支えに独立して慎重に進める。

- **PH2 — RAW-Trash を移植（実装済み）** ✅
  `FileManager` だけの純ファイル操作。`RawTrashCore` として実装（連番リネームで衝突退避、
  2 パス仕分け）、ユニットテスト 4 件通過。専用タブとして取り込みワークフロー先頭に据えた。

- **PH3 — 仕上げ**
  タブ間のフォルダ引き継ぎ UX、配布（Sparkle 自動更新・公証・配布サイト）を
  1本の .app に集約。GainForge の配布実績を継承する。

## 7. 動作環境・制約

- **macOS 15 以降**（GainForge の ISO ゲインマップ対応に合わせて最低要件を引き上げ）
- **Swift 6 / Xcode 16 以降**
- Apple Silicon 前提
- ビルドシステム: SwiftPM（Core / CLI）+ XcodeGen（App）
- 変換ロジックは各機能 Core に閉じ込め、共通シェルからは `convert(...)` 越しに呼ぶ

## 8. 将来の拡張候補

タブを増やす形で以下を受け入れられる設計にしておく（現スコープ外）:

- **ジオタグ付与**（GeoTagger 由来）— GPX トラックで撮影位置を EXIF に書き込み。
  移植の難所は exiftool 相当の EXIF/GPS 書き込みと、オフラインの座標→タイムゾーン変換。
- **重複掃除**（UniPic 由来）— pHash による類似・重複検出。sharp → ImageIO 置換が主。
