JPPhotoTools を MIT で公開し、GitHub Release による配布（Developer ID 署名・公証・Sparkle 自動更新）を運用するための手順書。

コード側の対応（`release.yml` / Sparkle 統合 / ExportOptions / ランディングページ）は実装済み。ここに書くのは、**あなた自身のアカウント情報に依存するため自動化できない一度きりのセットアップ**と、**毎回のリリース発行手順**の2つ。

前提: Apple Developer Program（有料）への加入。未加入の場合は末尾の「署名なし配布へ切り替える場合」を参照。

---

## 1. 一度だけ行うセットアップ

初回リリース前に一通り済ませる。以降のリリースでは不要。

### 1-1. Sparkle 署名鍵を生成する

Sparkle は appcast（更新情報）を EdDSA 署名で検証する。公開鍵をアプリに埋め込み、秘密鍵で CI が署名する。

```bash
# Sparkle 2.9.3 のツール一式を取得
curl -fL -o sparkle.tar.xz \
  https://github.com/sparkle-project/Sparkle/releases/download/2.9.3/Sparkle-2.9.3.tar.xz
mkdir -p sparkle-tools && tar -xf sparkle.tar.xz -C sparkle-tools

# 鍵ペアを生成（秘密鍵はキーチェーンに保存され、公開鍵が標準出力に出る）
./sparkle-tools/bin/generate_keys
```

出力される公開鍵（`SUPublicEDKey` 用の Base64 文字列）を控える。あわせて秘密鍵を書き出す:

```bash
# CI の Secret 登録用に秘密鍵をファイルへ書き出す
./sparkle-tools/bin/generate_keys -x sparkle_private_key.txt
```

> ⚠️ `sparkle_private_key.txt` は秘密鍵。**リポジトリにコミットしない**こと（登録後に削除してよい。キーチェーンに原本が残る）。

### 1-2. 公開鍵をアプリに埋め込む

[App/project.yml](../App/project.yml) の `SUPublicEDKey` を、生成した公開鍵に置き換える。

```yaml
# 置換前
SUPublicEDKey: "REPLACE_WITH_SPARKLE_ED_PUBLIC_KEY"
# 置換後（例）
SUPublicEDKey: "WiyUPbS5LnynuM2+J8Z/n+nRIhbjDpwtD/rbmcWeGVg="
```

Info.plist を再生成して反映する（`project.yml` が真実のソース）:

```bash
cd App && xcodegen generate
```

`App/JPPhotoTools-Info.plist` の `SUPublicEDKey` も同じ値に更新されていることを確認し、両ファイルをコミットする。

> `release.yml` は `REPLACE_WITH_SPARKLE_ED_PUBLIC_KEY` が残っていると即エラーで停止するガードを持つ。置換漏れの検知用。

### 1-3. Developer ID 証明書を用意する

Mac App Store 外で配布するには **Developer ID Application** 証明書が必要。

1. Xcode → Settings → Accounts、または [Apple Developer](https://developer.apple.com/account/resources/certificates) で証明書を作成。
2. キーチェーンアクセスで「Developer ID Application: 〜」を書き出し、`.p12`（パスワード付き）にする。
3. Base64 化する:

```bash
base64 -i DeveloperID.p12 | pbcopy   # クリップボードへ（Secret 登録に使う）
```

チーム ID は [App/ExportOptions.plist](../App/ExportOptions.plist) の `teamID` と Secret `TEAM_ID` で使う（Apple Developer の Membership 画面で確認できる 10 桁）。

### 1-4. 公証用 App Store Connect API キーを用意する

`notarytool` はパスワードより API キー方式が安定。

1. [App Store Connect → Users and Access → Integrations → App Store Connect API](https://appstoreconnect.apple.com/access/integrations/api) で API キーを作成（Role: Developer で可）。
2. ダウンロードした `AuthKey_XXXXXXXXXX.p8` を Base64 化:

```bash
base64 -i AuthKey_XXXXXXXXXX.p8 | pbcopy
```

3. **Key ID**（ファイル名の `XXXXXXXXXX`）と、同画面上部の **Issuer ID**（UUID）を控える。

### 1-5. GitHub Secrets を登録する

リポジトリの **Settings → Secrets and variables → Actions → New repository secret** で以下を登録する。

| Secret 名 | 中身 | 由来 |
|---|---|---|
| `TEAM_ID` | 10 桁のチーム ID | 1-3 |
| `DEVELOPER_ID_CERT_P12` | `.p12` の Base64 文字列 | 1-3 |
| `DEVELOPER_ID_CERT_PASSWORD` | `.p12` 書き出し時のパスワード | 1-3 |
| `ASC_API_KEY_P8` | `.p8` の Base64 文字列 | 1-4 |
| `ASC_KEY_ID` | API キーの Key ID | 1-4 |
| `ASC_ISSUER_ID` | Issuer ID（UUID） | 1-4 |
| `SPARKLE_PRIVATE_KEY` | 秘密鍵（`sparkle_private_key.txt` の中身） | 1-1 |

`GITHUB_TOKEN` は GitHub が自動発行するため登録不要。

### 1-6. GitHub Pages を有効化する

appcast とランディングページの配信先。

1. **Settings → Pages** を開く。
2. **Source** を `Deploy from a branch`、**Branch** を `gh-pages`（初回リリース後に自動生成される）、フォルダ `/ (root)` にする。
3. 公開 URL は `https://junpeiwada.github.io/JPPhotoTools/`。これが Info.plist の `SUFeedURL`（`.../appcast.xml`）と一致している必要がある。

> `gh-pages` ブランチは初回リリースのワークフローが自動作成する。初回はブランチ生成後にこの設定を行えばよい。

## 2. リリースを発行する（毎回）

セットアップ済みなら、リリースはタグ push だけで完結する。

```bash
# バージョンを決めてタグを打つ（vプレフィックス必須）
git tag v0.1.0
git push origin v0.1.0
```

これで [.github/workflows/release.yml](../.github/workflows/release.yml) が起動し、以下を自動実行する:

1. XcodeGen でプロジェクト生成 → Developer ID 署名でアーカイブ
2. `.app` をエクスポートし公証（notarytool）→ ステープル
3. 配布 zip を作成し **GitHub Release を作成**（zip を添付、リリースノートは自動生成）
4. `appcast.xml` を生成（秘密鍵で署名）し、ランディングページとともに **gh-pages へ publish**

- バージョン番号はタグから決まる（`v0.1.0` → `0.1.0`）。
- Sparkle が比較に使う `CFBundleVersion` は、単調増加する GitHub Actions の run 番号を自動採用する。
- 手動実行（Actions → Release → Run workflow）でもバージョンを入力して発行できる。

### 発行後の確認

- **Releases** に zip が添付されているか。
- `https://junpeiwada.github.io/JPPhotoTools/appcast.xml` が更新され、最新版を指しているか。
- 旧バージョンのアプリを起動 → メニュー「更新を確認…」または About 画面から更新が検出されるか。
- ダウンロードした `.app` を初回起動して Gatekeeper 警告が出ないか（公証・ステープルが効いていれば警告なしで開く）。

## 3. バージョン番号の運用

- `MARKETING_VERSION`（表示用バージョン）は **タグの値が優先**され、CI が上書きする。[App/project.yml](../App/project.yml) の `MARKETING_VERSION` はローカルビルド時の既定値。
- タグは必ず `v` プレフィックス付き（`v1.2.0`）。
- Sparkle は `CFBundleVersion`（= run 番号）で新旧を判定するため、リリースのたびに単調増加する。手動で管理する必要はない。

## 署名なし配布へ切り替える場合（Apple Developer 未加入）

Apple Developer Program に加入しない場合、署名・公証・Sparkle 自動更新は使えない。その場合は `release.yml` を「ビルド → `.app` を zip 化 → Release に添付」だけに簡略化する（証明書・公証・Sparkle 関連ステップと Secret を削除）。

- ユーザーは初回起動時に **右クリック →「開く」** で Gatekeeper を回避する必要がある（未署名のため）。
- 自動更新は無効。更新は Releases ページから手動でダウンロードしてもらう。
- この切り替えが必要になったら、その旨を伝えてもらえれば `release.yml` と Sparkle 統合を最小構成に落とす。
